# Convex Guide for This Project

This guide explains how Convex is used in the Slack clone and how to work with it.

## What Is Convex?

Convex is a **backend-as-a-service** that gives you:

1. **A database** – Tables (like `users`, `workspaces`, `channels`, `messages`) with a defined schema.
2. **Serverless functions** – **Queries** (read-only, reactive) and **Mutations** (write data).
3. **Real-time** – When data changes, every subscribed client (e.g. your Next.js app) gets updated automatically.

You don’t run your own server. You write functions in the `convex/` folder; Convex runs them and syncs data to the client.

---

## Setup (if you haven’t already)

1. **Install dependencies**

   ```bash
   pnpm install
   ```

2. **Log in and create a Convex project**

   ```bash
   pnpm convex:dev
   ```

   - Opens browser to log in at [convex.dev](https://convex.dev).
   - Creates (or selects) a Convex project and a **deployment**.
   - Locally, Convex:
     - Pushes your `convex/` code to that deployment.
     - Generates `convex/_generated/` (including `api.js` and types).
     - Writes `.env.local` with `NEXT_PUBLIC_CONVEX_URL`.

3. **Run the app**

   ```bash
   pnpm dev
   ```

The app uses `NEXT_PUBLIC_CONVEX_URL` to connect to your Convex backend.

## Project Layout Under `convex/`

| File | Role |
|------|------|
| `schema.ts` | Defines tables and indexes. All data must match this. |
| `users.ts` | Create/get/list/update users. |
| `workspaces.ts` | Create/get/list/update/delete workspaces; add members. |
| `channels.ts` | Create/get/list/update/delete channels; join channel. |
| `messages.ts` | Send/list/update/delete **channel** messages. |
| `directMessages.ts` | Get-or-create DM thread; send/list/delete **DM** messages. |
| `_generated/` | **Do not edit.** Generated by `convex dev` / `convex codegen`. |

## Core Concepts

### 1. Schema (`convex/schema.ts`)

Tables are defined with `defineTable` and validators from `convex/values` (`v.string()`, `v.id("tableName")`, etc.):

```ts
defineTable({
  name: v.string(),
  createdBy: v.id("users"),
  // ...
}).index("by_workspace", ["workspaceId"])
```

- Every table gets `_id` and `_creationTime` automatically.
- Indexes (e.g. `by_workspace`) are used for fast queries like “all channels in this workspace.”

### 2. Queries (read, reactive)

- **Read-only.** They read from the database and return data.
- **Reactive:** When data they depend on changes, Convex re-runs them and pushes new results to the client.

Example:

```ts
// convex/channels.ts
export const listByWorkspace = query({
  args: { workspaceId: v.id("workspaces") },
  handler: async (ctx, args) => {
    return await ctx.db
      .query("channels")
      .withIndex("by_workspace", (q) => q.eq("workspaceId", args.workspaceId))
      .collect();
  },
});
```

In React you use `useQuery`:

```ts
const channels = useQuery(api.channels.listByWorkspace, { workspaceId });
// channels updates automatically when backend data changes
```

### 3. Mutations (write)

- **Change data:** insert, patch, delete.
- Called from the client (e.g. on button click) with `useMutation`.

Example:

```ts
// convex/messages.ts
export const send = mutation({
  args: {
    channelId: v.id("channels"),
    userId: v.id("users"),
    body: v.string(),
    userName: v.string(),
    userImageUrl: v.string(),
  },
  handler: async (ctx, args) => {
    return await ctx.db.insert("messages", { ... });
  },
});
```

In React:

```ts
const sendMessage = useMutation(api.messages.send);
await sendMessage({ channelId, userId, body, userName, userImageUrl });
```

After the mutation runs, any active `useQuery` that depends on that data (e.g. `api.messages.listByChannel`) will get new results automatically.

## How This App Uses Convex

1. **Auth (demo)**  
   No real auth. On “Join,” we create a user in Convex and store their `_id` in `localStorage`. That Id is used as “current user” for all mutations and queries.

2. **Workspaces**  
   - Create: `workspaces.create` + add creator to `workspaceMembers`.  
   - List: `workspaces.listForUser` by current user.  
   - Update/delete: `workspaces.update` / `workspaces.remove` (delete also cleans members/channels/messages).

3. **Channels**  
   - Create: `channels.create` (and add creator to `channelMembers`).  
   - List: `channels.listByWorkspace`.  
   - When you open a channel, we call `channels.join` so you’re a member.  
   - Edit/delete: `channels.update` / `channels.remove` (with UI confirm for delete).

4. **Channel messages**  
   - Send: `messages.send`.  
   - List: `messages.listByChannel` (reactive).  
   - Edit/delete: `messages.update` / `messages.remove`.

5. **DMs**  
   - Start: `directMessages.getOrCreateThread` for two users in a workspace.  
   - List threads: `directMessages.listThreadsForUser`.  
   - Send: `directMessages.send`.  
   - List messages: `directMessages.listByThread`.  
   - Delete: `directMessages.remove`.

All of these are wired in the UI so every button and CRUD path works.

## Useful Commands

| Command | What it does |
|--------|------------------|
| `pnpm convex:dev` | Run Convex dev: syncs `convex/` to your deployment, watches for changes, generates `_generated/`. |
| `pnpm convex:deploy` | Deploy Convex to production (e.g. after `pnpm build`). |

Always run `convex dev` at least once before `pnpm dev` so `_generated/api.js` exists and the app can call Convex.

- Use it to:
  - See tables and data.
  - Inspect logs for queries/mutations.
  - Manage deployments and env vars.

## Summary

- **Schema** in `convex/schema.ts` defines tables and indexes.
- **Queries** = read + reactive; **Mutations** = write.
- **React:** `useQuery(api.module.functionName, args)` and `useMutation(api.module.functionName)`.
- **Real-time:** No extra code; Convex pushes query results when data changes.
- **First-time setup:** Run `pnpm convex:dev`, then `pnpm dev` with the generated `.env.local`.

For more, see [Convex docs](https://docs.convex.dev).
